# 3.3 接入 OpenAI

## 背景与目标

OpenAI 和 Anthropic 都支持工具调用，但它们的消息格式**不兼容**。这是本节最核心的内容——理解这个差异，才能理解 Provider 抽象的价值。

**本节结束后，你应该有：**
- `src/providers/openai.ts`：`OpenAIProvider` 类，实现 `LLMProvider` 接口
- `bin/index.ts` 里的 `createProvider()` 支持 `STEVEN_PROVIDER=openai`
- 两个 Provider 用同一套 Agent Loop 代码，对外行为一致

---

## 核心差异：消息格式

这是实现 OpenAI Provider 最难的部分——把我们内部的 `Message[]` 格式转成 OpenAI API 接受的格式。

### 工具调用的 assistant 消息

**Anthropic 格式（我们的内部格式）：**
```json
{
  "role": "assistant",
  "content": [
    { "type": "text", "text": "我来读一下这个文件" },
    { "type": "tool_use", "id": "toolu_01", "name": "Read", "input": { "file_path": "/tmp/a.ts" } }
  ]
}
```

**OpenAI 格式：**
```json
{
  "role": "assistant",
  "content": "我来读一下这个文件",
  "tool_calls": [
    {
      "id": "call_abc123",
      "type": "function",
      "function": {
        "name": "Read",
        "arguments": "{\"file_path\":\"/tmp/a.ts\"}"
      }
    }
  ]
}
```

关键差异：
- OpenAI 的 `content` 是字符串，不是数组
- OpenAI 的工具调用在 `tool_calls` 字段里，不在 `content` 数组里
- OpenAI 的 `arguments` 是**JSON 字符串**，不是对象

### 工具结果消息

**Anthropic 格式（我们的内部格式）：**
```json
{
  "role": "user",
  "content": [
    { "type": "tool_result", "tool_use_id": "toolu_01", "content": "文件内容...", "is_error": false }
  ]
}
```

**OpenAI 格式：**
```json
{
  "role": "tool",
  "tool_call_id": "call_abc123",
  "content": "文件内容..."
}
```

关键差异：
- OpenAI 工具结果是**独立的 `role: "tool"` 消息**，不是嵌套在 `user` 消息里
- 字段名：`tool_call_id`（OpenAI）vs `tool_use_id`（Anthropic）
- OpenAI 没有 `is_error` 字段（错误信息直接放在 `content` 里）

**消息格式转换：**

把我们的 `Message[]` 转成 `OpenAI.ChatCompletionMessageParam[]`：

```ts
function toOpenAIMessages(messages: Message[]): OpenAI.ChatCompletionMessageParam[] {
  const result: OpenAI.ChatCompletionMessageParam[] = []
  
  for (const msg of messages) {
    if (msg.role === 'user') {
      // user 消息：普通文本 or 工具结果
      const toolResults = msg.content.filter(b => b.type === 'tool_result')
      const textBlocks = msg.content.filter(b => b.type === 'text')
      
      if (toolResults.length > 0) {
        // 工具结果：每个 tool_result 变成独立的 role: "tool" 消息
        for (const block of toolResults) {
          result.push({
            role: 'tool',
            tool_call_id: block.tool_use_id,
            content: block.content,
          })
        }
      } else {
        // 普通 user 消息
        result.push({
          role: 'user',
          content: textBlocks.map(b => b.text).join(''),
        })
      }
    } else if (msg.role === 'assistant') {
      // assistant 消息：文本 + 工具调用
      const text = msg.content
        .filter(b => b.type === 'text')
        .map(b => b.text)
        .join('')
      const toolUses = msg.content.filter(b => b.type === 'tool_use')
      
      result.push({
        role: 'assistant',
        content: text || null,
        tool_calls: toolUses.length > 0
          ? toolUses.map(b => ({
              id: b.id,
              type: 'function' as const,
              function: {
                name: b.name,
                arguments: JSON.stringify(b.input),
              },
            }))
          : undefined,
      })
    }
  }
  
  return result
}
```

**工具格式转换：**

```ts
function toOpenAITools(tools: Tool[]): OpenAI.ChatCompletionTool[] {
  return tools.map(tool => ({
    type: 'function' as const,
    function: {
      name: tool.name,
      description: tool.description,
      parameters: z.toJSONSchema(tool.inputSchema),
    },
  }))
}
```

**流式处理：**

OpenAI 的流式 chunks 结构和 Anthropic 不同，但我们 yield 出来的 `LLMStreamEvent` 类型一致：

```ts
// 在 call() 里
const stream = await this.client.chat.completions.create({
  model,
  messages: [{ role: 'system', content: systemPrompt }, ...toOpenAIMessages(messages)],
  tools: toOpenAITools(tools),
  stream: true,
})

// 工具调用需要在内部累积（OpenAI 也是流式返回 tool_calls）
const toolCalls: Map<number, { id: string; name: string; args: string }> = new Map()

for await (const chunk of stream) {
  const delta = chunk.choices[0]?.delta
  if (!delta) continue

  // 文本增量
  if (delta.content) {
    yield { type: 'text_delta', text: delta.content }
  }

  // 工具调用增量（按 index 累积）
  if (delta.tool_calls) {
    for (const tc of delta.tool_calls) {
      const existing = toolCalls.get(tc.index) ?? { id: '', name: '', args: '' }
      toolCalls.set(tc.index, {
        id: existing.id || tc.id || '',
        name: existing.name || tc.function?.name || '',
        args: existing.args + (tc.function?.arguments ?? ''),
      })
    }
  }

  // 流结束时 emit 完整的 tool_use 事件
  if (chunk.choices[0]?.finish_reason === 'tool_calls') {
    for (const [, tc] of toolCalls) {
      yield {
        type: 'tool_use',
        id: tc.id,
        name: tc.name,
        input: JSON.parse(tc.args),
      }
    }
  }
}
```

### 更新 `bin/index.ts`

在 `createProvider()` 里加入 OpenAI 分支：

```ts
import { OpenAIProvider } from '../src/providers/openai.ts'

function createProvider(): LLMProvider {
  const providerName = process.env.STEVEN_PROVIDER ?? 'anthropic'
  
  switch (providerName) {
    case 'anthropic':
      return new AnthropicProvider()
    case 'openai':
      return new OpenAIProvider()
    default:
      throw new Error(`Unknown provider: ${providerName}. Supported: anthropic, openai`)
  }
}
```

```bash
STEVEN_PROVIDER=openai OPENAI_API_KEY=sk-... bun run bin/index.ts -p "列出当前目录的文件"

```

对比两个 Provider 的结果：

```bash
# 同一个 prompt，两个 Provider
ANTHROPIC_API_KEY=sk-ant-... bun run bin/index.ts -p "搜索所有包含 AsyncGenerator 的文件"
STEVEN_PROVIDER=openai OPENAI_API_KEY=sk-... bun run bin/index.ts -p "搜索所有包含 AsyncGenerator 的文件"
```

两者应该找到相同的文件（可能回复措辞不同，但工具调用结果一致）。
