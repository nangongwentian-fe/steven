# 3.1 LLMProvider 接口

## 背景与目标

抽象的关键问题是：**边界画在哪里？**

我们的 Agent Loop 和 LLM Provider 之间，有两个可能的边界位置：

**方案 A：在 query.ts 内部对每个 Provider 分支**
```ts
// query.ts 内部
if (provider === 'anthropic') {
  yield* callAnthropic(messages, tools, ...)
} else if (provider === 'openai') {
  yield* callOpenAI(messages, tools, ...)
}
```

**方案 B：定义接口，Provider 实现接口，query.ts 只调接口**
```ts
// query.ts 内部
yield* params.provider.call(messages, tools, ...)
```

方案 A 每加一个 Provider 就要改 query.ts，永远无法终止；方案 B 是真正的"开闭原则"——对扩展开放，对修改关闭。

**本节结束后，你应该有：**
- `src/core/provider.ts`：`LLMStreamEvent` 类型 + `LLMProvider` 接口
- 更新后的 `src/core/query.ts`：`QueryParams` 包含 `provider` 字段，循环体调用 `provider.call()`

---

## Claude Code 是怎么做的

打开 `references/claude-code/src/Tool.ts`，找 `AnyTool` 的定义；再打开 `references/claude-code/src/query.ts`，看 `queryLoop()` 接受什么参数。

Claude Code 有多个 Provider（Anthropic、Bedrock、Vertex、Foundry），它们共享同一个 `claude.ts` 里的调用层，但对外暴露统一的接口。接口边界在"如何调用 LLM"，不在"LLM 叫什么名字"。

## 动手实现

### 第一步：定义 `LLMStreamEvent` 和 `LLMProvider`

创建 `src/core/provider.ts`：

**`LLMStreamEvent`** — Provider 在流式调用过程中 yield 出的事件，`query.ts` 消费这些事件：

```ts
export type LLMStreamEvent =
  | { type: 'text_delta'; text: string }
  | { type: 'tool_use'; id: string; name: string; input: Record<string, unknown> }
```

注意：这两种事件和 `callAnthropic()` 目前 yield 的内容**完全一致**——重构后 `query.ts` 不需要任何改动，只是把 `callAnthropic()` 换成 `provider.call()`。

**`LLMProvider`** — 所有 Provider 必须实现的接口：

```ts
export interface LLMProvider {
  call(
    messages: Message[],
    tools: Tool[],
    systemPrompt: string,
    model: string,
  ): AsyncGenerator<LLMStreamEvent>
}
```

**inputSchema：** 和第一章的 `callAnthropic()` 签名完全一致，只是换成接口方法。这意味着后续你可以把 `src/anthropic.ts` 里的 `callAnthropic()` 函数体几乎原封不动地搬进 `AnthropicProvider.call()`。

---

### 第二步：更新 `QueryParams`

在 `src/core/query.ts` 里，给 `QueryParams` 加一个 `provider` 字段：

```ts
export interface QueryParams {
  messages: Message[]
  tools: Tool[]
  systemPrompt: string
  model: string
  provider: LLMProvider  // ← 新增
  maxTurns?: number
}
```

同时把 loop 体内的 `callAnthropic(messages, tools, systemPrompt, model)` 改为 `params.provider.call(messages, tools, systemPrompt, model)`。

---

### 第三步：更新 `Engine` 和 `bin/index.ts`

**`src/core/engine.ts`：**

`Engine` 构造函数接受 `provider: LLMProvider`，创建 `QueryParams` 时把它传进去：

```ts
export class Engine {
  constructor(
    private tools: Tool[],
    private provider: LLMProvider,
    // ... 其他参数
  ) {}
}
```

**`bin/index.ts`：**

根据环境变量选择 Provider，传给 Engine：

```ts
const provider = createProvider()  // 你来实现这个工厂函数
const engine = new Engine(ALL_TOOLS, provider, ...)
```

`createProvider()` 的逻辑：读取 `STEVEN_PROVIDER` 环境变量（默认 `'anthropic'`），返回对应的 Provider 实例。

---

## 验证

这一步不需要写新功能，运行现有测试验证重构没有破坏任何东西：

```bash
bun test src/
```

```bash

```

行为应该和重构前完全一致。
