# 1.2 类型系统

## 背景与目标

Agent 和 LLM 之间的所有通信，都是**消息（Message）的来回传递**。在写任何逻辑之前，我们需要先定义好这些消息的"形状"。

**本节结束后，你应该有：**
- `src/types/message.ts`：完整的消息类型体系
- `src/types/tool.ts`：工具接口的类型定义

---

## LLM 对话的本质结构

和 LLM 的对话是一个**消息列表**，每条消息有 `role`（谁说的）和 `content`（说了什么）：

```
[
  { role: "user",      content: "帮我读一下 src/index.ts" },
  { role: "assistant", content: [{ type: "tool_use", name: "Read", ... }] },
  { role: "user",      content: [{ type: "tool_result", ... }] },
  { role: "assistant", content: "文件内容是..." },
]
```

注意 `content` 有两种形式：
- **字符串**：普通文本消息
- **ContentBlock 数组**：包含工具调用、工具结果、文本块的混合内容

---

## Claude Code 是怎么做的

打开 `references/claude-code/src/types/message.ts`。

你会看到 Claude Code 的消息类型系统非常复杂（有 `UserMessage`、`AssistantMessage`、`SystemMessage`、`ProgressMessage` 等十几种类型），这是因为它还要处理 UI 渲染、流式事件等场景。

**我们只需要 API 通信所需的核心部分：**

| 我们需要 | Claude Code 对应 |
|---------|----------------|
| `Message` | `UserMessage` / `AssistantMessage` |
| `TextBlock` | `BetaTextBlock` |
| `ToolUseBlock` | `BetaToolUseBlock` |
| `ToolResultBlock` | `BetaToolResultBlockParam` |

---

## 动手实现

### `src/types/message.ts`

需要定义以下类型：

**`TextBlock`** — 纯文本内容块：
- `type: 'text'`
- `text: string`

**`ToolUseBlock`** — LLM 决定调用工具时产生：
- `type: 'tool_use'`
- `id: string` — 工具调用的唯一 ID（用于匹配后续的 tool_result）
- `name: string` — 工具名称
- `input: unknown` — 工具的输入参数

**`ToolResultBlock`** — 工具执行结果，作为下一轮用户消息发回给 LLM：
- `type: 'tool_result'`
- `tool_use_id: string` — 对应哪次工具调用
- `content: string` — 执行结果内容
- `is_error?: boolean` — 可选，执行失败时为 `true`

**`ContentBlock`** — 以上三种的联合类型

**`Message`** — 最终的消息类型：
- `role: 'user' | 'assistant'`
- `content: string | ContentBlock[]`

### `src/types/tool.ts`

需要定义：

**`ToolResult`** — 工具执行的返回值：
- `content: string`
- `isError?: boolean`

**`Tool<TInput>`** — 工具的接口：
- `name: string`
- `description: string`
- `inputSchema: z.ZodObject<...>` — 用 Zod 定义输入 schema
- `isReadOnly: boolean` — 是否只读（影响并发策略，第二章详解）
- `call(input: TInput): Promise<ToolResult>`

**`ToolDef<TInput>`** — 传给 `defineTool()` 工厂的类型，`isReadOnly` 设为可选。

---

## 关键设计决策

**为什么 `ToolResultBlock` 里的 `content` 是 `string` 而不是 `unknown`？**

Anthropic API 要求 `tool_result` 的 `content` 必须是字符串（或字符串数组）。工具的执行结果，无论多复杂，都需要序列化为字符串传回给 LLM。

**为什么 `Tool` 的 `inputSchema` 用 Zod 而不是手写 JSON Schema？**

Zod schema 可以自动生成 JSON Schema（发给 API）和做运行时类型校验（验证 LLM 返回的参数）。一份定义，两种用途。Claude Code 也是这样做的——查看 `references/claude-code/src/Tool.ts` 的 `inputSchema: Input` 字段，`Input extends AnyObject = AnyObject` 就是 Zod 的 `ZodObject`。

## 验证

在 `src/types/message.ts` 里写一个简单的测试（Bun 支持直接运行 ts 文件）：

```ts
// 在文件底部临时添加，验证完删掉
const msg: Message = {
  role: 'user',
  content: [
    { type: 'tool_result', tool_use_id: 'toolu_01', content: 'hello' }
  ]
}
console.log(msg) // 能编译运行说明类型定义正确
```
