# 1.4 Agent Loop

## 背景与目标

这是整个 Coding Agent 最核心的部分。**Agent Loop** 驱动了"思考 → 执行 → 观察"的循环，是 Agent 能自主完成任务的根本原因。

**本节结束后，你应该有：**
- `src/core/query.ts`：完整的 Agent Loop，使用 AsyncGenerator 实现流式事件
- 能用一个 mock 工具验证循环逻辑跑通

---

## Agent Loop 的本质

一次 Agent 对话不是简单的"问 → 答"，而是一个**循环**：

```
用户输入
  ↓
调用 LLM（携带工具定义）
  ↓
LLM 返回 tool_use？
  ├── 是 → 执行工具 → 把结果追加到消息 → 重新调用 LLM
  └── 否 → 输出最终回复 → 结束（或等待下一轮用户输入）
```

关键点在于：**LLM 可能连续调用多个工具**，每次调用后，工具结果作为新的 `user` 消息追加进对话历史，再次调用 LLM，直到 LLM 不再需要工具为止。

---

## 为什么用 AsyncGenerator？

我们需要**边执行边输出**——文本流式显示、工具执行实时反馈。如果用 `Promise<string>`，只能等所有事都做完才能返回，用户体验很差。

AsyncGenerator 让我们可以：
```ts
for await (const event of query(params)) {
  if (event.type === 'text') process.stdout.write(event.text)
  if (event.type === 'tool_start') console.log(`Running ${event.name}...`)
}
```

这和 Claude Code 的设计完全一致。

## Claude Code 是怎么做的

打开 `references/claude-code/src/query.ts`，找到 `queryLoop()` 函数（大约在第 241 行）。

注意几个关键设计：

**1. 状态对象 `State`（第 204 行）**
```ts
type State = {
  messages: Message[]
  toolUseContext: ToolUseContext
  // ...其他字段
}
```
所有跨迭代的可变状态都放在一个对象里，每次循环开头解构，`continue` 点整体赋值新 state。这避免了跨迭代的隐式状态突变。

**2. `while(true)` 循环（第 307 行）**
不用 `for` 循环，因为退出条件不是计数器，而是"LLM 不再返回工具调用"。

**3. 工具并发（第 98 行 `runTools()`）**
`toolOrchestration.ts` 里，只读工具并发执行，写工具串行执行。我们第一章先不实现并发，第二章加工具时再考虑。

## 动手实现

创建 `src/core/query.ts`，实现核心的 `query()` AsyncGenerator 和辅助的 `runTools()` 函数。

### 需要定义的类型

**`QueryEvent`** — query() yield 出的事件：
- `{ type: 'text'; text: string }` — 文本增量
- `{ type: 'tool_start'; toolUseId: string; name: string; input: unknown }` — 工具开始执行
- `{ type: 'tool_result'; toolUseId: string; name: string; result: ToolResult }` — 工具执行完毕
- `{ type: 'done' }` — 本轮 Agent 执行完毕

**`QueryParams`** — query() 的入参：
- `messages: Message[]` — 初始消息历史
- `tools: Tool[]` — 可用工具列表
- `systemPrompt: string`
- `maxTurns?: number` — 最大循环次数，防止无限循环（建议默认 20）

### `query()` 的实现逻辑

```
let messages = [...params.messages]

for turn 0..maxTurns:
  1. 调用 callAnthropic(messages, tools, systemPrompt)
  2. 遍历流式事件：
     - text_delta → yield { type: 'text', text }
     - tool_use   → 收集到 toolUseBlocks[]
  3. 把本轮 LLM 响应追加到 messages（assistant 消息）
  4. 如果 toolUseBlocks 为空 → yield { type: 'done' } → return
  5. 执行所有工具（调用 runTools()）
     - 每个工具执行前 yield tool_start
     - 每个工具执行后 yield tool_result
  6. 把工具结果追加到 messages（user 消息，内容是 ToolResultBlock[]）
  7. 继续下一轮

yield { type: 'done' }  // maxTurns 用完
```

### `runTools()` 的实现逻辑

第一章先实现**串行版本**（简单，后续扩展）：

```
for each toolUseBlock:
  找到对应的 Tool
  如果找不到 → 返回 { content: 'Unknown tool: xxx', isError: true }
  调用 tool.call(block.input)
  如果抛出异常 → 返回 { content: 错误信息, isError: true }
```

> 工具执行失败**不应该抛出异常**，而是把错误作为 `tool_result` 传给 LLM，让 LLM 决定如何处理。这是 Claude Code 的一贯做法。

### mock tool 用于测试

先写一个 mock 工具验证循环逻辑，不依赖真实文件系统：

```ts
const echoTool: Tool<{ msg: string }> = {
  name: 'Echo',
  description: 'Echoes the message back',
  inputSchema: z.object({ msg: z.string() }),
  isReadOnly: true,
  async call(input) {
    return { content: `echo: ${input.msg}` }
  }
}
```
